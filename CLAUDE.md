## Token pipeline

Figma variables ŌåÆ JSON snapshots in `packages/cli/snapshots/` ŌåÆ CLI generators ŌåÆ outputs in `packages/design-tokens/dist/` (`tokens.css`, `text-styles.css`, `tailwind.preset.cjs`) and `packages/design-tokens/src/generated/tokens.ts`.

Commands (run from repo root):
- `pnpm tokens:fetch` ŌĆö pull fresh variables from Figma via WebSocket bridge (requires Figma desktop + Desktop Bridge plugin)
- `pnpm tokens:generate` ŌĆö regenerate outputs from snapshots
- `pnpm tokens:check` ŌĆö diff regenerated output vs committed (CI uses this)
- `pnpm tokens:sync` ŌĆö fetch + generate

**Bridge port note**: `tokens:fetch` binds a WebSocket *server* on port `9226` (fallbacks `9227ŌĆō9229`) and waits for the Figma Desktop Bridge plugin to dial in. The figma-console MCP runs the *same* kind of server.

**Why `pnpm tokens:fetch` (and `tokens:sync`) fails inside a Claude Code session ŌĆö confirmed 2026-05-27:** The Desktop Bridge plugin holds **exactly one connection at a time**. If any Claude Code session has the figma-console MCP connected, the plugin is bonded to *it* (the plugin panel shows "MCP ready"), so the CLI's competing server never wins the plugin:
- ports `9226`/`9227` ŌåÆ `skip` (plugin never dials in; 8s timeout each)
- a later port may complete a stale WS handshake, but `EXECUTE_CODE` comes back in the MCP's envelope shape, not the CLI's expected `{id, result:"<json>"}` ŌåÆ fails with `"[object Object]" is not valid JSON` and file shows `"?"`.

This is **structural**: `tokens:fetch` (CLI bridge) and `figma_execute` (figma-console MCP) are mutually-exclusive consumers of one single-connection plugin. Closing the *previous* session doesn't help if the *current* one also has the MCP ŌĆö it re-grabs the plugin instantly.

**Two ways to refresh tokens:**
1. **Plain terminal, no Claude session with the MCP** ŌĆö quit all Claude Code sessions running figma-console MCP, then `pnpm tokens:sync` in a normal terminal. (Can't be driven by Claude, since Claude needs the MCP.)
2. **From inside a Claude session (the practical path)** ŌĆö the session already owns the plugin via the MCP, so refresh snapshots by running the CLI's extraction snippets (`packages/cli/src/bridge/extract.ts`: `variableCollectionSnippet`, `textStylesSnippet`, `effectStylesSnippet`) through `figma_execute`, write each returned JSON string verbatim to `packages/cli/snapshots/<file>.json`, then run `pnpm tokens:generate` + `pnpm tokens:check`. The 5 collections are Brand/Alias/Mapped Colors (leading space in Figma)/Mapped Type/Mapped Sizes. This is exactly what the figma-design-system-sync skill's Phase 2a now documents.

## Per-component workflow

Every component goes through the 8-step workflow defined in [PLAN.md](PLAN.md#per-component-workflow):

1. **Behaviour spec** ŌĆö read Figma description + annotations + variants. Synthesize: what it is, how it works, when to use, when NOT to use. Draft missing pieces. **Confirm with user before writing code.**
2. Property mapping (Figma variants ŌåÆ React props)
3. Implement in `packages/components/src/<Name>/`
4. Storybook story covering every variant
5. Lint: zero hardcoded colors/sizes
6. Visual diff against Figma at multiple breakpoints
7. **Parity check** with `figma_check_design_parity` ŌĆö gate on **visual fidelity only**:
   - Zero `critical` discrepancies anywhere
   - Zero `major` discrepancies in `visual` / `spacing` / `typography` categories
   - Documented numeric / normalization artifacts (pill radius `896` vs `9999`, COMPONENT_SET chrome `134├Ś80` vs `40├Ś40`) explicitly allowed ŌĆö note in the PLAN.md sign-off line
   - Aggregate score is **informational only**, not a gate. React-composition idioms cause `componentAPI` deductions that are paradigm choices, not visual gaps. Ceilings: simple atoms 90ŌĆō96, composition-heavy components (Buttons) 60ŌĆō70, components with COMPONENT_SET chrome 75ŌĆō85.
8. Mark done in PLAN.md

Key rules:
- **Behaviour spec confirmation is non-negotiable** ŌĆö never start implementation until the user has approved the spec.
- **Never assume a variant's structure** ŌĆö component sets routinely vary *structurally* across variants (different layout direction, different child elements, different trailing content, present-but-hidden nodes). Analyze **every** variant individually: walk each one's tree and check node `visible` flags **and stroke/fill paint `visible` ŌĆö not just stroke weight**. Do not extract one variant and extrapolate to the rest. (Learned 2026-06-04: Desktop Progress Indicator's Upload type is horizontal with a trailing %/check while Task & System Process are vertical with a helper-text line and no %; the Stepper count "underline" was a 1.5px stroke whose paint was `visible:false` in every state except Focus.)
- **Always VIEW the rendered variant image during the build, not just the structure tree** ŌĆö download `figma_get_component_image` to a temp file and `Read` it. The structure tree hides visual patterns the code must reproduce ŌĆö e.g. Pagination's page-range windowing (`1 2 3 ŌĆ” N`), exact active-cell styling, real glyph shapes. (Learned 2026-06-05: built Pagination's page range from the tree alone; the windowing didn't match Figma's `1 2 3 ŌĆ” 10` until the rendered image was viewed.)
- **Respect nested-instance property configuration** ŌĆö when a component (molecule/organism/template/etc.) composes an *instance* of another component (an atom/molecule), the new component must reproduce the **exact property values that instance carries in Figma**, not the consumed component's React defaults. Read each nested instance's `componentProperties` from the Figma tree and pass them through explicitly; never assume the atom's default matches. (Learned 2026-06-08: Breadcrumbs' Plain Button instances are `Show Underline: false` in Figma ŌĆö and the `Underline` node is `visible:false` ŌĆö but the `Button` atom defaults `showUnderline = true`, so the crumbs shipped with an underline the design doesn't have until `showUnderline={false}` was passed.)
- **Visual parity is non-negotiable** ŌĆö zero `critical`, zero `major` in `visual` / `spacing` / `typography` before checking off. Aggregate parity score is informational.
- **Figma Slots ŌåÆ composable child-injection props (never flatten, never hardcode)** ŌĆö Figma natively supports **Slots** (`node.type === 'SLOT'`), used across the complex components (Modal Headers, Content Primitives, Content Cards, Cells, etc.) as flexible composition points. They are NON-NEGOTIABLE to detect and map correctly:
  1. **Detect** ŌĆö during property-mapping, walk the component tree for **every** `type === 'SLOT'` node (in every variant). For each slot capture: its **name**, its **auto-layout** (`layoutMode`/`itemSpacing`/padding/`primaryAxisAlignItems`/`counterAxisAlignItems`/`layoutSizingHorizontal`/`layoutSizingVertical`), its **visibility**, and any **Preferred Instances**. List all detected slots in the behaviour spec presented to the user before building.
  2. **Map** ŌĆö each named Figma slot ŌåÆ a `ReactNode` slot prop named from the slot's Figma name (e.g. "Secondary Content" ŌåÆ `secondaryLeading`/`secondaryTrailing` when the slot's auto-layout visibly splits the row; otherwise one prop). The slot's auto-layout becomes a thin wrapper `<div>` the component owns; the **contents are always injected by the caller**. Never hardcode slot contents into the base component. Items set to "Fill Container" in Figma ŌåÆ `width:100%`/`flex:1`.
  3. **Type** ŌĆö if the slot declares Preferred Instances, document them in the prop's JSDoc as the permitted-components contract.
  4. **Visibility mirrors Figma** ŌĆö keep an explicit `show<SlotName>` boolean toggle (so prototypes can reveal/animate the slot later), but its **default** mirrors the Figma slot's `visible` flag: visible slot ŌåÆ defaults `true` and reserves its row even when empty; hidden slot ŌåÆ defaults `false` but the prop still exists for prototype use. **Do NOT render invented dashed-outline placeholders** ŌĆö if the slot is visible in Figma, render it with the real nested instances Figma shows; if hidden, render nothing (just reserve the prop). (Learned 2026-06-09: Modal Headers' "Secondary Content" SLOT `9200:38903` is a SPACE_BETWEEN row ŌĆö Status Badges left, Header CTAs right ŌĆö that ships visible; mapped to `secondaryLeading`/`secondaryTrailing` with `showSecondaryContent` defaulting `true`, default story mirrors Figma's Scheduled badge + 2 Dispatch buttons.)

## Figma MCP strategy

Use the Figma Dev MCP first; escalate to figma-console MCP for richer data, writes, or parity checks. Detail in [PLAN.md ┬¦ Tooling Notes](PLAN.md#figma-access-strategy).

Common calls:
- `figma_get_component_for_development` ŌĆö variants + token bindings + screenshot
- `figma_get_component` ŌĆö description + property definitions
- `figma_get_annotations` ŌĆö designer-authored specs
- `figma_check_design_parity` ŌĆö required for component sign-off
- `figma_execute` ŌĆö escape hatch for ad-hoc plugin API calls (resolving variable IDs, refreshing snapshots without the bridge)

### figma-console REST token lives in `~/.claude.json` (confirmed 2026-06-03)

REST-based calls (`figma_check_design_parity`, `figma_take_screenshot`, `figma_get_component_image`) authenticate with `FIGMA_ACCESS_TOKEN`. **Claude Code reads its MCP `env` from `~/.claude.json` (`mcpServers.figma-console.env.FIGMA_ACCESS_TOKEN`), NOT from `claude_desktop_config.json`** ŌĆö those are separate files for separate apps (the latter feeds the Claude Desktop chat). A `403 {"err":"Token expired"}` despite a valid PAT almost always means the token was updated in the wrong file, or the live MCP server is running a stale env. The harness **caches MCP env at session start** and reuses it even when the server is killed/respawned mid-session ŌĆö so after editing `~/.claude.json` a **full Claude Code session restart** is required (a mid-session `/mcp` reconnect or `kill` re-spawns with the cached old env). Verify the token directly with `curl -H "X-Figma-Token: $TOK" https://api.figma.com/v1/me` (200 = valid) and `ps eww <mcp-pid> | tr ' ' '\n' | grep FIGMA_ACCESS_TOKEN` to see which token the live server actually holds. Keep both config files in sync when rotating the PAT.

### Why the bridge sometimes prompts to publish (confirmed 2026-06-03)

The Desktop Bridge plugin does **not** silently edit components/variables ŌĆö code inspection shows its only launch-time action is a read (`figma.loadAllPagesAsync()` to attach a `documentchange` *listener*); every document-writing call (`setProperties`, `setBoundVariable`, `editComponentProperty`, `componentPropertyDefinitions =`) is gated behind an explicit edit/generation tool. **Read-only sessions do not flip the file's modified flag** ŌĆö verified 2026-06-03: a read-only session (status probes, `figma_execute` reads, screenshots) left the Library clean with no publish prompt. So every Publish prompt reflects a *real* change to a published asset. The catch: Figma counts **layout / organizational moves** (reparenting or reordering published components ŌĆö e.g. the icon-frame category reorg) as publishable changes even when no property or variable changed ŌĆö that's the usual "but I didn't change anything" surprise. Mitigations: batch edits and publish once; open the Publish dialog (it lists exactly what changed) and dismiss/publish accordingly; for purely experimental or organizational work, use a separate scratch file so the published Library stays stable.

## Storybook conventions

- Component stories live next to the component: `packages/components/src/<Name>/<Name>.stories.tsx`
- Title: `Atoms/<Name>` (or `Molecules/`, `Organisms/`, etc.)
- Stories cover every Figma variant + a `Catalog` story showing the full matrix
- Foundations stories live in `apps/storybook/src/foundations/` and auto-derive from the generated tokens map
- **CSS grid Catalog stories:** Every component instance that is a direct child of a CSS grid must have `style={{ justifySelf: 'start' }}` (or be wrapped in a `<div>`). Without it the component stretches to fill the grid cell instead of hugging its content. Apply to all future Catalog stories.
- **Flex column layout:** When building stories or layouts with `flexDirection: 'column'`, flex containers default to `align-items: stretch`. If children should hug their natural width, add `alignItems: 'flex-start'` to the wrapper.
- **Story-list confirmation is mandatory (added 2026-05-25):** Before creating a Storybook story file for any component, present the planned list of stories to the user via `AskUserQuestion` and wait for explicit approval. Allowed story kinds: (a) a Figma variant of the component, (b) a boolean property of the component, (c) the default state, (d) a single-component `Catalog` / `AllStates` matrix view. **Never invent multi-instance "showcase" scenarios** (filter groups, wizard flows, tab bars, delivery-context grids, removable lists, etc.) without prior approval ŌĆö those are molecule-level compositions and live elsewhere in the design system.
- **Sidebar ordering:** Story navigation order in the sidebar is controlled by `apps/storybook/.storybook/preview.tsx` ŌåÆ `storySort.order`. The Atoms section currently forces ViewSwitcherTab ŌåÆ SegmentSwitcherTab ŌåÆ PageTab ŌåÆ WizardTab ŌåÆ Title to be adjacent, with everything else sorted alphabetically via `'*'`.

## Focus state conventions

All interactive components use the same `:focus-visible` treatment:

```css
outline: var(--stroke-sm) solid var(--border-secondary-component-focus);
outline-offset: 4px;
```

`--stroke-sm` = 1.5px; focus color = yankees-blue-900 (light) / yankees-blue-50 (dark). Mobile Button currently shares this with Desktop in code ŌĆö the proper Mobile Focus variant is pending in Figma, parity will be re-checked once that's added.

## Map views ŌĆö standard map primitive (set 2026-06-24)

**Every map view in the system renders through `MapView`** (`packages/components/src/MapView/MapView.tsx`) ŌĆö the canonical interactive map primitive. Do **not** hand-roll maps (no OSM `<iframe>` embeds, no static screenshots) in new components; compose `<MapView>` and override `center`/`zoom`/`tileUrl` as needed.

- **Engine:** [Leaflet](https://leafletjs.com/) (`leaflet` dependency, `@types/leaflet` dev) with **OpenStreetMap** raster tiles ŌĆö open-source, **no API key**. Leaflet is loaded client-side via dynamic `import('leaflet')` (SSR-safe) and externalized in the tsup build (`--external leaflet`); its CSS is injected at runtime (CDN-pinned, matching the runtime-style-injection convention ŌĆö no build-time CSS import). Requires network access for tiles.
- **Zoom:** Leaflet's built-in zoom control is disabled (`zoomControl: false`); the design-system `<MapZoomControl>` is overlaid bottom-right and wired to `map.zoomIn()` / `map.zoomOut()`, disabling at min/max zoom.
- **This is interim.** It stays the standard **until a more robust provider (e.g. Google Maps) is adopted** ŌĆö when that happens, swap the internals of `MapView` only and every consumer (Extra Large Modal, future map screens) updates automatically. Keep the public `MapView` API provider-agnostic.

2. **Map every region to its `@leta/components` component** (Page ŌåÆ SideBar/TopBar/TableContainer ŌåÆ
   Table ŌåÆ Cell types ŌåÆ DesktopDropdowns/DateTimePicker, etc.).
3. **Compose only.** Raw `<div>`/`<button>` with token styles that *imitate* a component (a custom
   "table", a hand-built menu, a restyled chip) is a **bug**. Raw elements are allowed ONLY for
   non-component layout primitives ŌĆö flex wrappers, portals, positioning shells.
4. If a region appears to have **no** matching component, **STOP and confirm with the user** before
   hand-rolling anything.

**Why this rule exists:** the first Orders/Deliveries screen was hand-rolled (custom HeaderCell/
BodyCell/OrderRow, a fake table, hand-built dropdowns) because it was built from a *screenshot
mental-model* instead of the wireframe's instance tree. Bridge-inspecting `594:59213` showed it is
literally `SideBar + TopBar + Data Table Page Body (pad [24,24,0,24]) + Content (pad-bottom 16) +
TableContainer(Default) ŌåÆ Table ŌåÆ Cell` ŌĆö all already built. Always inspect instances first.

### Reading flow connections from the wireframe (set 2026-06-29)

How the designer encodes how components connect, and how to read each via the bridge:

1. **Prototype interactions (the arrows)** ŌĆö the PRIMARY signal. Read `node.reactions`
   (`trigger.type` like `ON_CLICK`/`ON_HOVER`/`MOUSE_ENTER` + `action.destinationId`). These give the
   exact destination node ID for "clicking/hovering X opens Y" ŌĆö machine-readable, no text parsing.
   The Scheduled section has ~26 of these (e.g. row Ōŗ» ŌåÆ overflow menu, avatar ŌåÆ user menu, Created ŌåÆ
   date picker). Resolve `destinationId` to a node to see the target component/group.
2. **Pinned annotations (the Annotate tool)** ŌĆö readable via `figma_get_annotations` or `node.annotations`
   (`label`/`labelMarkdown`). The designer names a group descriptively (e.g. "User Menu Dropdown") and
   references it by name in an annotation on the trigger; read both and connect them.
3. **Sticky-note widgets** ŌĆö their text is **NOT** extractable via the plugin API (`widgetSyncedState`
   returns `{}`). **Screenshot the node** (`figma_capture_screenshot <id>`) and read the image instead.

**MANDATORY before building any flow:** after reading the prototype reactions + annotations + sticky
notes, **present the inferred connections back to the user as a multiple-choice quiz / short list of
clarifying questions (AskUserQuestion) and wait for confirmation** before implementing. Do not assume a
connection's exact behavior (e.g. whether a status filter defaults to a sub-status or shows the whole
group) ŌĆö verify it.

**Token-name gotcha (confirmed 2026-06-29):** there is **no `--surface-neutral-bg-primary`** token ŌĆö
white surfaces use **`--surface-neutral-bg-default`** (others: `-muted`/`-raised`/`-subtle`). A
`var(--surface-neutral-bg-primary)` reference resolves to nothing ŌåÆ transparent (caused a dropdown
bleed-through bug). CLAUDE.md notes above that say "bg-primary" are stale; the real token is `bg-default`.
After editing any `packages/components` source, **rebuild the dist** (`pnpm --filter @leta/components build`)
or the playground (which imports the built dist) shows the old version ŌĆö and **clear Vite's cache**
(`rm -rf apps/playground/node_modules/.vite`) + restart so the pre-bundled dep updates.
